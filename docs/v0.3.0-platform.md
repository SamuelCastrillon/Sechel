# Sechel v0.3.0 — Platform-Agnostic MCP Server

## Por qué existe

v0.1.0 (rebranding) y v0.2.0 (panel admin) asumen **Next.js + Vercel** como
plataforma de despliegue única. Eso funcionó para el prototipo, pero nos ata a:

- **Vercel** para edge deploy (no podemos deployar en Cloudflare Workers,
  Railway, Fly.io, ni un VPS liviano).
- **Next.js** como dependencia runtime incluso cuando el producto real es el
  handler MCP, no el panel admin.
- **300 MB+ de node_modules** para servir un handler HTTP con SQLite.

v0.3.0 extrae el MCP server de Next.js para que viva como un **paquete
independiente** que deploya en *cualquier* runtime (Node, Bun, Deno, CF
Workers) con un `Dockerfile` de ~30 MB, y publica un **CLI bridge** npm que
cualquiera instala y usa sin deployar nada.

## Stack target

| Capa | v0.2.x (hoy) | v0.3.0 |
|---|---|---|
| Framework HTTP | Next.js (App Router) | **Hono** (Node, CF Workers, Deno, Bun) |
| Panel admin | Next.js + Server Actions | **Preact** + Vite + REST API |
| MCP handler | `mcp-handler` sobre Next.js | `mcp-handler` sobre Hono |
| Core domain (`modules/core/`) | `server-only` + Next imports | **TS puro** sin framework |
| CLI | No existe | **`sechel`** CLI bridge stdio→HTTP |
| Deploy primario | Vercel | **Cualquier edge + Docker + VPS** |
| Imagen Docker | N/A (Next.js build ~300 MB) | ~30 MB |
| Publicación npm | No | `@sechel/core`, `sechel` CLI |

## Plataformas objetivo

```
Target runtimes (todos con el mismo código fuente):

  Node.js 20+          → pnpm start (prod), VPS, Railway, Fly.io
  Cloudflare Workers   → wrangler deploy
  Deno                 → deno run
  Bun                  → bun run
  Vercel               → sigue funcionando (Hono tiene adapter Vercel)
```

## Cambios con respecto a v0.2.x

### 1. Core domain: sacar dependencia de Next.js

**Hoy:** `modules/core/` tiene `import 'server-only'` en ~6 archivos,
y resuelve paths relativos a `process.cwd()` pensando en Next.js.

**v0.3.0:** el paquete `@sechel/core` es TS puro:

- Eliminar `import 'server-only'` de todos los archivos de core.
- Reemplazar `getDb()` singleton con una factory explícita
  (`createDb(options)`), sin singleton global — así el mismo código corre
  en serverless (cada request crea/obtiene su pool) y en VPS (long-lived).
- Mover la lógica de `resolveUrl()` + `getEnv()` a opciones de factory
  en vez de `process.env` directo.

```
// Antes (hoy)
import { getDb } from '@/modules/core/db'
const db = await getDb()       // singleton global

// Después (v0.3.0)
import { createDb } from '@sechel/core'
const db = createDb({ url, token })  // fresco por request o singleton según runtime
```

### 2. MCP handler: de Next.js route a Hono

**Hoy:** `app/api/mcp/route.ts` importa `handler` de `modules/mcp/server.ts`.
El handler usa `withMcpAuth(createMcpHandler(...), verifyToken)` y funciona
solo como Next.js route handler.

**v0.3.0:** el mismo `createMcpHandler` se envuelve con Hono:

```typescript
// packages/mcp-server/src/adapter/hono.ts
import { Hono } from 'hono'
import { handler } from '@sechel/mcp-server'
import { verifyToken } from '@sechel/core'

export function createSechelApp() {
  const app = new Hono()

  // MCP endpoint — el mismo handler que hoy
  app.all('/api/mcp', async (c) => {
    const authInfo = await verifyToken(c.req.raw)
    const response = await handler(c.req.raw, { authInfo })
    return response
  })

  // REST API para el panel
  app.route('/api/admin', adminApi)

  // Static files del panel (build de Preact)
  app.get('/admin/*', serveStatic({ root: './panel/dist' }))

  return app
}

// Node
serve(createSechelApp())

// CF Workers
export default createSechelApp()

// Vercel
export const handler = createSechelApp().fetch
```

### 3. Panel: de Next.js Server Actions a Preact SPA + REST

**Por qué Preact y no Vue/Svelte:** Preact es React-compatible (3 KB vs
40 KB). Los componentes del panel actual se migran casi sin cambios —
JSX igual, hooks igual, estados igual. Lo que cambia:

- Server Actions → `fetch()` a REST endpoints del Hono server.
- `next/navigation` → `preact-router` o `wouter`.
- `next/cookies` → localStorage: token JWT se guarda en el cliente.
- Build: `vite build` produce `dist/` estático.

El panel build-eado se sirve desde el Hono server como archivos estáticos
o deploya aparte (Netlify, CF Pages).

### 4. CLI bridge: stdio → HTTP

El paquete `sechel` en npm expone un binario que hace MCP en modo **stdio**
y proxea cada tool call a un Sechel remoto via HTTP.

```
$ pnpm add -g sechel
$ sechel config set url https://mi-instance.vercel.app/api/mcp
$ sechel config set token sek_abc123...
$ sechel    ← arranca en modo stdio, habla con el agente local
```

**Cómo funciona:**

1. El agente local envía `tools/call` con `{ name: "mem_save", arguments: {...} }`
   por stdio.
2. El bridge recibe, hace `POST https://instance/api/mcp` con el mismo payload,
   y agrega `Authorization: Bearer <token>`.
3. Devuelve la respuesta por stdio. El agente no nota la diferencia.

Implementación trivial:

```typescript
// packages/cli/src/bridge.ts
import { StdioServerTransport } from '@modelcontextprotocol/sdk/server/stdio.js'
import { McpServer } from '@modelcontextprotocol/sdk/server/mcp.js'

const server = new McpServer({ name: 'sechel-bridge' })
const { url, token } = readConfig()

// Registrar todas las tools mem_* como proxy
const TOOLS = ['mem_save', 'mem_search', 'mem_context', /* ... */]
for (const tool of TOOLS) {
  server.registerTool(tool, { /* schema from remote */ }, async (args) => {
    const res = await fetch(`${url}/api/mcp`, {
      method: 'POST',
      headers: { Authorization: `Bearer ${token}`, 'Content-Type': 'application/json' },
      body: JSON.stringify({ method: 'tools/call', params: { name: tool, arguments: args } }),
    })
    return res.json()
  })
}

const transport = new StdioServerTransport()
await server.connect(transport)
```

### 5. Server local embedido

El CLI también puede arrancar un Sechel completo **local**:

```
$ sechel start
  ✓ Sechel corriendo en http://localhost:3030
  ✓ MCP endpoint: http://localhost:3030/api/mcp
  ✓ Panel admin: http://localhost:3030/admin
  ✓ DB: SQLite local (./.sechel.db)
```

Esto es el `@sechel/mcp-server` corriendo con `@libsql/client` apuntando a
un archivo SQLite local. Sin Turso, sin Vercel, sin deploy. Ideal para:

- Desarrollo local sin depender de Turso remoto.
- Equipos chicos que no quieren pagar Turso.
- Pruebas offline.

### 6. Docker + VPS

```dockerfile
# apps/server/Dockerfile
FROM node:20-alpine AS build
WORKDIR /app
COPY pnpm-lock.yaml package.json pnpm-workspace.yaml ./
COPY packages/ ./packages/
RUN corepack enable && pnpm install --frozen-lockfile --filter @sechel/server
COPY apps/server/ ./apps/server/
RUN pnpm --filter @sechel/server build

FROM node:20-alpine
WORKDIR /app
COPY --from=build /app/apps/server/dist ./dist
COPY --from=build /app/node_modules ./node_modules
EXPOSE 3000
CMD ["node", "dist/index.js"]
```

```yaml

  sechel:

      - sechel-data:/data

  sechel-data:
```

## Riesgos y mitigaciones

| Riesgo | Impacto | Mitigación |
|---|---|---|
| `@libsql/client` no funciona en CF Workers | Alto | Usar `@libsql/client/worker` (export específico para edge); probar en fase P-1 |
| Hono no soporta Streamable HTTP tan bien como Next.js | Medio | `mcp-handler` es agnóstico del framework; el adapter Hono es un wrapper mínimo |
| Migrar panel a Preact rompe funcionalidad existente | Medio | El panel de Next.js sigue funcionando durante P-6; se reemplaza cuando el nuevo esté probado |
| Pérdida de Server Actions (streaming de UI) | Bajo | El panel admin no necesita streaming; REST es suficiente |
| Aumento de complejidad del monorepo | Medio | `pnpm workspace` + `packages/` claro; CI build por paquete |

## Criterios de éxito

- [ ] El MCP server deploya en Cloudflare Workers y responde `mem_save`/`mem_search`.
- [ ] El MCP server deploya en un VPS vía Docker y responde igual.
- [ ] `npm install -g sechel` + `sechel` como MCP command funciona en OpenCode/Claude.
- [ ] `sechel start` levanta server completo con SQLite local.
- [ ] `pnpm test` pasa igual en Next.js (v0.2.x) y en standalone (v0.3.0).
- [ ] El panel admin funciona como SPA estática servida desde Hono.
- [ ] La imagen Docker pesa < 100 MB.

## Backward compatibility con v0.2.x

- `app/api/mcp/route.ts` sigue funcionando — el handler es el mismo, solo
  cambia cómo se sirve.
- El panel de Next.js coexiste con el nuevo panel SPA durante la migración.
- `@sechel/core` tiene la misma API que `modules/core` — el código de
  herramientas MCP no cambia.
- Las migraciones de base de datos no cambian (schema idéntico).
- `.env` vars no cambian (las mismas, mismas defaults).
